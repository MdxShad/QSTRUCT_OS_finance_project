Frontend for Q‑Struct OS
========================

This directory contains a minimal React/TypeScript single‑page application built with Vite.  The frontend communicates with the FastAPI backend to parse RFQs, analyse quote quality, generate counterfactual alternatives and display audit records.  Styling is implemented with Tailwind CSS for a dark, professional theme.

## Setup

Ensure you have Node 18 or later and `npm` installed.  Then install dependencies:

```bash
cd frontend
npm install
```

## Running in development

Start the Vite development server:

```bash
npm run dev
```

Vite will launch a local server (typically on port 5173) and proxy API requests beginning with `/api` to `http://localhost:8000`, where the FastAPI backend should be running.  Open your browser at `http://localhost:5173` to load the application.

## Building for production

To create an optimised production build, run:

```bash
npm run build
```

The compiled static assets will be output to the `dist/` directory.  You can preview the build locally with:

```bash
npm run preview
```

## Overview of components

- **RFQInput** – A textarea for entering free‑form RFQ descriptions, along with buttons to parse or optimise.  A seed button populates the demo RFQ.
- **PayoffObjectPanel** – Displays the parsed payoff object in a table, showing only fields that are populated.
- **QuoteQualityCards** – Shows headline metrics such as win probability, desk economics score and overall risk score.
- **AlternativesTable** – Lists counterfactual alternatives generated by the backend, sorted by final score.  The recommended alternative is highlighted.
- **ExplanationPanel** – Provides a narrative explanation of why the recommended alternative improves the quote.
- **AuditTrace** – Shows a simple list of past optimisation calls with timestamps and the recommended alternative ID.

All API calls are made via the helper functions defined in `src/api/client.ts` using the Fetch API.

## Notes

This frontend is intentionally simple.  It does not perform advanced error handling, does not save state between refreshes, and does not implement authentication.  The primary goal is to visualise the synthetic optimisation results produced by the backend.
